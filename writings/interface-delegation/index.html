Interface Delegation

Interface delegation is a common concept in programming languages like Cobol and Delphi. “Delegation refers to one object relying upon another to provide a specified set of functionalities” — (wikipedia), but in our physical world is way more common than we could imagine.

Lets see an example:

The Sky remote inherits features from the set-to-box device providing the user physical access to complete tasks and perform actions.

Meanwhile the interface on the TV provides feedback about what you should do with your remote to interact with it.

But why this matters in design?

Interfaces are extensions of our capabilities, either physical or cognitives. They allow us to execute a really wide range of tasks in a more practical or effective way. They could be physical ( like the remote ) or virtual ( like the GUI on the screen ).

Interface delegation as “one object relying upon another to provide a specified set of functionalities” means we will have two ( or more ) interfaces to interact with.

But sometimes this situation always ends up in something like this:

The “grandma” paradigm

The “grandma” paradigm is nothing but answer to the basic question: “how will user actually use this”.

Our job as designers should always derive from the exhaustive analysis of data. User research, user journeys, interaction analysis, etc.

Most of “grandma” nightmares born when we “delegate interfaces” from one device to another in order “to provide more control” or “solve every single user scenario”

But lets see another example:

The Apple remote feature only two main blocks of inputs, a touch surface and an array of action buttons that controls the main interface ( this one includes a voice recognition interface trough a microphone )

Both interfaces seems to be quite different between each other ( despite the fact they are not designed to do exactly the same ) Apple interfaces are usually quite simple and straightforward quite contrary to the typical embedded solution used in most set-top-boxes, but why?!

Research, Design, Testing, Research, Design, Testing…

The TL;DR version is pretty simple, Apple invest on research.

The long answer involves vision, respect and methodology, but I’m not going to address that subject now.

Ok, but how it works?

Well, in Design, interface delegation helps users to extend or concentrate their interaction outside the main device, allowing them to be independent from it.

A non-digital example could be the “classic manual stick transmission” in cars, where the user literally moves the gears from one position to another in order to transfer the energy from the motor to the wheels. Controlling the action trough a direct interface. Versus the “modern sync paddle shift” where the user controls an interface that controls a computer who process the action to move the transmission gears in synchronic order.

How they differ? well they do different things. Although the objective of both is the same: to transfer the movement of the motor to the wheels, how they work internally, in terms of mechanics, technology, efficiency, design and experience, make them absolutely different solutions.

It delegate the processing of how the mechanical parts work to a more simple and sophisticated interface that allow us only to change gears ups and down. You no longer need to push the clutch, move the stick to the gear, push the gas, release the clutch. You just push up or down, the car does the rest.

In the digital world we can see this all the time. Smartwatches are secondary devices acting as a delegated surrogates who allow us to control the main interface, our phones, but even if the most advanced ones, that have more functions that don’t need the phone to work are basically delegated interfaces.

Ive got it, so when to use it?

When you need it!

No, seriously, Interface delegation is a concept, a tool. It will only makes sense when your research shows you you need to give the user that solution. Using it depends on the correct implementation of your design, result of the correct evaluation of the data you collected on the research stages of your product development.
